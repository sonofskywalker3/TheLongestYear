# TLY 0.12.0 — Clarity & Balance — Design Spec

**Date:** 2026-06-15
**Status:** Approved for planning
**Scope:** First of three post-beta releases. See the roadmap note below for 0.13.0 / 0.14.0.

## Purpose

The beta drew consistent, friendly feedback (Nexus posts + Reddit thread, scraped 2026-06-14). The two loudest signals:

1. **Confusion about the Junimo-Point / weekly-theme economy.** Many testers (TheFirstBanana, Dusklight7, emmainthealps, Tutorem, xsansara) didn't understand that you should *not* rush to complete bundles — you bank more JP and avoid the weekly downside by holding off-theme items until that theme comes up. They also reported the festival "noon return" as a bug when it is deliberate.
2. **The run is too easy — finishable in year 1 with zero resets.** khauser13, xsansara, PokeTheSilver204, Tutorem all hit this. Root causes are the Traveling Cart handing you rare donation items so you stockpile a chest, and run-trivializing perks. The deep fix (owned bundles + difficulty tiers) is 0.13.0; 0.12.0 takes the first real bite via the cart-slot economy.

0.12.0 is the "ships in days" release that relieves the live confusion and lands the first balance lever, while the big engine work is specced separately.

## Out of scope (decided, do not build here)

- **Owned bundles, per-loop re-randomization, Easy/Normal/Hard tiers** → 0.13.0.
- **Seasonal "retaliation" escalations** → 0.14.0.
- **Festival noon-return (#3)** → WORKING-AS-INTENDED. `FestivalTimeFlow` deliberately preserves your real clock so leaving a festival early doesn't waste the day. No code change; documented in the onboarding letter.
- **Ancient-seed recipe/seed lost between loops (#5)** → WORKING-AS-INTENDED. The recipe is wiped like any non-banked recipe (`FarmerReset.cs:160`) and the seed by the inventory wipe (`FarmerReset.cs:34`); the museum rebuilds empty each loop (TLY never touches it) so the reward is re-earnable by re-donating. Banking it in the Craftbook is an optional convenience. No code change.

## Deliverables

### 1. First-loop onboarding letter

A single mailbox letter delivered on **Spring 1 of the first loop only**.

- **Gate:** `MetaState.CompletedResets == 0` (first loop) AND not already received this run. Guard with `RunActivation.IsActive` like every TLY feature so it never fires on a non-TLY save.
- **Delivery:** add a `Data/Mail` entry via the SMAPI asset-edit pattern already used by `CommunityCenterUnlock` / `MountainUnlock`, then drop it into the mailbox on Spring 1 day-start (or `mailForTomorrow` on save creation). Reuse the existing mail plumbing — do not introduce a new mechanism.
- **Content (copy, plain and friendly):**
  - One line on what the mod is (loop year 1 until the Community Center is done; spend Junimo Points between loops).
  - **JP scales up the deeper you get** — later seasons award far more JP, so don't feel behind early.
  - **The big clarifier:** *Don't rush to complete bundles.* Hold off-theme items and wait for that theme's week — donating then banks the **1.5× bonus** AND lifts the week's downside. This is the single most-misunderstood mechanic.
  - One line that **festivals intentionally give your day back** — you can leave and re-enter, and leaving early keeps your clock, so returning "early" is a feature, not a bug.
- **Also:** tighten the weekly-theme quest description text (`WeeklyThemeQuestService`) to echo the "wait for the theme to bank the bonus + lift the downside" framing, so the guidance is visible in-game beyond the one-time letter.

### 2. Cart-slot economy

Throttle the Traveling Cart so you can't trivially stockpile every donation item.

- **Mechanic (visible-stock cap):** the cart **displays only N purchasable items** per visit. Start at **1**; buy additional slots with JP up to a cap of **10** (vanilla's full stock size).
- **Upgrades:** add `cart_slot_2 … cart_slot_10` (or an equivalent tiered entry) to `UpgradeCatalog`, banked in `MetaState.OwnedUpgrades` like every other upgrade. **Escalating JP cost** — cheap for the first extra slots, pricey for the last — so widening the cart is a real, ongoing investment rather than a one-time tax.
- **Implementation:** trim the traveling-merchant stock to the unlocked slot count where the cart's `ShopMenu` / stock dictionary is built (the only `TravelingMerchant` touch-point today is `CartStockPreview.cs`; the live shop will need a patch on stock construction). Trim deterministically so the same day shows a stable subset.
- **Forward note:** 0.13.0's difficulty tiers will set the *starting* slot count per tier (Easy starts wider, Hard starts at 1). 0.12.0 ships a single flat starting value of 1.

### 3. Bug #1 — "5 quality crops" clears on a single donation

**Report (khauser13, 11 Jun):** a weekly/bundle requirement asking for 5 quality crops is marked satisfied after donating one (1× parsnip clears it; reproduced with melons).

**Suspected root cause:** donation tracking keys on **item-id presence, not the slot's required stack quantity.** The live `DonationObserver` records an id into the `donated` set the moment it witnesses a deposit, and the gate/quest checks `donated.Contains(id)` — so a slot needing 5 reads satisfied at 1. Note `CcDonationReconciler` already derives donations from the game's **authoritative per-slot completion** (`completed[i]`), which *does* respect quantity; the live path does not.

**Fix direction:** make the "satisfied" determination respect quantity — prefer deferring to the game's per-slot completion state (as the reconciler does) rather than the live id-set, or track per-id donated counts against the slot's required count. Keep the change **minimal** — 0.13.0 rewrites this whole area when bundles become mod-owned, so do not refactor the donation pipeline here.

**Process:** this is a real bug with a hypothesis, not a confirmed cause. Reproduce first (systematic-debugging), confirm the live-vs-reconciled discrepancy, then fix under TDD. Do not ship a guess.

## Testing / verification

- **Onboarding letter:** new TLY run → letter present in mailbox Spring 1; start a 2nd loop → letter does NOT re-send (`CompletedResets > 0`); non-TLY save → nothing fires.
- **Cart slots:** fresh run shows 1 cart item; buying `cart_slot_N` widens the displayed stock by one each; cap holds at 10; slots persist across loops (banked upgrade).
- **Bug #1:** a 5-quantity quality-crop requirement reads 1/5 after one donation and only completes at 5/5; verify both the live in-menu display and the day-end gate agree.
- Engineering correctness (wiring fires, gated on `RunActivation`, no leak onto non-TLY saves) is verified solo; reserve a playtest only if the cart-slot feel needs human judgment.
