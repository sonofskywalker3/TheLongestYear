## Change 1 — Remove the donation-JP double-pay (bug fix)

**Today:** every donation banks JP live (`DonationService.OnItemDonated`,
`src/TheLongestYear/Donations/DonationService.cs:60`), and then `RunController.AwardInterimJp`
(`src/TheLongestYear/Loop/RunController.cs:940`) re-awards a full batch over ALL
`Run.DonatedItemIds` at run end — called on FailReset (line 586) and first Win (line 605), at
the run-end week's season multiplier. Every donated item is paid twice; a deliberate early
reset gets the whole run's donations re-paid on the way out, which *amplifies* the reset-farming
cheese.

**Change:** delete `AwardInterimJp` and both call sites. Live per-donation banking is the one
true payment. Update/delete the tests that assert the interim award.

**Player-facing effect:** run income drops roughly by the size of the old end-of-run batch. The
checkpoint award (Change 2) is the designed replacement income, pointed at progression instead
of resets.

## Change 2 — Season-checkpoint JP award (anti-cheese, user pick)

**New JP source:** passing a season checkpoint (the day-28 gate on Spring/Summer/Fall —
`RunAction.AdvanceMonth` branch of the day-end switch in `RunController.OnDayEnding`) pays a
one-time bonus,
scaled by the season multiplier of the season being ENTERED, so the message is "moving forward
pays better than looping spring":

| Checkpoint passed | Multiplier used | Award (base 100) |
|---|---|---|
| Spring → Summer | 1.5 | 150 JP |
| Summer → Fall | 2.5 | 250 JP |
| Fall → Winter | 4.0 | 400 JP |

- New `JpSettings.CheckpointCompletionBonus` (default **100**), config-tunable like the other
  JP constants. `jp_boost` applies (it applies to every JP source).
- Winter 28 is the win, not a checkpoint — no extra award there (the win flow already pays).
- Log at Info like bundle/room bonuses; message via i18n.
- No award on the Fail branch, obviously. Once per season boundary per run (the Continue branch
  only fires once per boundary by construction).

## Change 3 — XP-multiplier upgrades (user design)

khauser13: re-grinding levels every loop is the tedious part, keep-XP prices feel high for the
value. The user's design: multipliers make re-leveling fast *without* anyone being level 10 on
day 1, and professions get re-picked naturally on level-up nights. Keep-level chain is untouched
— keeps set your starting floor, multipliers set your slope; they complement.

**New catalog rows (Efficiency category, same idiom as jp_boost):**

- Per-skill chains `xp_mult_<skill>_1..4` for farming / fishing / foraging / mining / combat.
  Tier N multiplies that skill's XP gains: tier 1 = **x2**, 2 = **x3**, 3 = **x4**, 4 = **x5**.
  Chained via PrerequisiteId. Costs per skill: **100 / 200 / 350 / 550** (sum 1,200 per skill;
  all five chains = 6,000). Tunable at plan time.
- Capstone `xp_mult_all`, single tier, cost **3,000**, MetaRequirement = owning tier 4 in ALL
  five skills. Effect: doubles ALL skill XP **on top** of the per-skill tier (→ effective x10),
  and **also doubles Mastery XP** (khauser13 called out the mastery re-grind specifically).
  Per-skill tiers do NOT touch Mastery; only the capstone does. `keep_mastery` chain untouched.

**Implementation shape:** Harmony prefix on `Farmer.gainExperience(int which, int howMuch)`
scaling `howMuch` by the owned multiplier; gated on `RunActivation.IsActive` like every patch.
Mastery-XP doubling hooks wherever mastery exp actually accrues (verify the 1.6 path against
the decompile at plan time — do not assume it routes through gainExperience). Pure tier→factor
math lives in Core with unit tests; the patch is a thin shell.

## Change 4 — High-impact upgrade repricing: DEFERRED, decision gated

PokeTheSilver204 banked 750 JP in run 1 (under the double-pay economy) and bought
`cult_red_cabbage`, deleting the only hard blocker from run 2. **The user has NOT ruled on new
prices** — ruling: "depends on how well our remixed bundles do at keeping those two from being
the only genuinely hard items."

**This spec ships NO price changes.** The decision trigger is recorded here: after the
owned-bundle engine's baseline is playtested, revisit prices for the audit list —
`cult_red_cabbage` (750), `cult_starfruit` (750), `keep_bus` (1500), `fortune_rare_fish` (525).
Note the double-pay removal is itself an effective price increase (income drops), so measure
before touching numbers.

## Change 5 — Clarity / messaging

1. **Advertise JP season scaling in live UI.** The onboarding letter mentions it, but Dusklight7
   still "went gung ho" front-loading donations. Add a line to the weekly hub / planning view
   showing the CURRENT season's multiplier, e.g. "Junimo gratitude grows with the seasons —
   donations now earn x2.5 JP." (exact copy at plan time; i18n key; "Selected" not "championed"
   applies to all new strings). Optional flavor: Junimo line "each day we get stronger!"
   (Dusklight7's suggestion) if a natural surface exists — do not build a new surface for it.
2. **Empty-week message.** Correction from plan-time code reading: a HUD message already
   exists (`hud.nothing-to-donate`: "Nothing left to donate for this theme - drawback
   lifted.", shown by `RunController.ApplyEmptyPoolLiftIfNeeded`). The change is a REWORD to
   Dusklight7's warmer flavor: "The Junimos are overwhelmed by your donations - no requests
   this week! Drawback lifted."
3. **Checkpoint award visibility.** The Change-2 award gets a clear on-screen/log line — it IS
   the anti-cheese messaging ("progressing pays").

All new strings land in `i18n/default.json` (guard tests enforce this since 0.11.55).

  drained theme.
