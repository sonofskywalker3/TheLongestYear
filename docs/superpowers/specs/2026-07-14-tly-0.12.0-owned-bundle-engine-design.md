# TLY 0.12.0 — Spec 2 of 2: Owned-Bundle Engine — Design

**Status:** APPROVED-IN-BRAINSTORM 2026-07-14 (supersedes the WIP doc
`2026-06-15-tly-0.13.0-A-owned-bundle-engine-design.md`; pulled forward from 0.13.0-A into
0.12.0 by user direction); awaiting written-spec review.
**Milestone:** 0.12.0 = the economy/clarity spec (build first) + this engine.

## What it is

TLY stops reading the bundles the game generated and **writes its own bundle set** into the
save — at run creation and inside every loop reset. Same 6 rooms, same themes
(`RoomThemeMap`), same bundles-per-room counts as vanilla; what re-rolls every loop is **which
bundles appear**, **what items they ask for**, and **their stack/quality asks**. The CC menu,
donations, and room restoration use our bundles exactly as if vanilla had made them.

**Why:** (a) strong players can't run the same script twice — this is the difficulty fix for
"three players finished year 1 without looping"; (b) the read-and-classify path
(`BundleCatalogBuilder` guessing what each remixed bundle means) dies, and with it the whole
remix-fragility bug class — bundles missing from checkpoints, khauser13's premature win,
impossible weekly themes.

## Locked mechanics (ruled 2026-07-10, restated)

- **Write mechanism:** TLY builds the `BundleData` strings itself and calls
  `Game1.netWorldState.Value.SetBundleData(...)`. NOT vanilla's `GenerateBundles` over an edited
  `Data/RandomBundles` (its seeding and schema don't fit). The write path + downstream re-read
  were live-proven 2026-07-10 (`debug ShuffleBundles` + `tly_classify`, 0 errors).
- **Seed:** `BundleEngineSeed.For(Game1.player.UniqueMultiplayerID, MetaState.CompletedResets)`
  (repo XOR+prime idiom). **AMENDED at implementation (v0.11.75, reviewer-verified):** the spec's
  original `uniqueIDForThisGame` basis is unusable — `Utility.NewUniqueIdForThisGame` is
  wall-clock-derived and the reset RE-ROLLS it, so neither the pre- nor post-reset value survives
  a replay/reload consistently. The farmer's `UniqueMultiplayerID` is assigned once at creation,
  persisted, and untouched by `loadForNewGame` — do NOT "correct" this back. Reloading or
  replaying a reset regenerates IDENTICAL bundles — no bundle-reroll save-scum. Generation
  happens ONLY at run-create and inside the reset.
- **Reset sequence:** extend `CommunityCenterUnlock.Apply()`: generate → `SetBundleData` →
  clear stale `netWorldState.Bundles`/`BundleRewards` → `areasComplete` all false →
  `cc.MakeMapModifications(force:true)` → re-reveal the 6 Junimo notes. One ordered sequence.
- **Migration:** none. An in-flight loop finishes on its existing bundles; the first reset after
  updating generates fresh. The read-from-game classify path is then deleted, not maintained.
- **Downstream:** `BundleCatalogBuilder` keeps reading `netWorldState.BundleData`, but the
  generator ALSO emits each bundle's requirement manifest (kind, quota ramp, stacks, qualities)
  directly — classification consults the manifest first; nothing is guessed.
  `CcDonationReconciler` untouched.

## Generation model — "expanded-pool remix"

For each room: draw that room's bundle count from its pool (vanilla's bundles for that room +
our authored ones below), then fill each bundle's slots from its item pool.

### Item pools — derived from live game data (SVE-proof by construction)

Pools are built at generation time from the game's OWN data tables, so any mod-added content
(SVE crops, fish, forage, etc.) is included automatically without TLY ever having seen it:

| Pool | Source | Notes |
|---|---|---|
| Seasonal crops | `Data/Crops` (plantable + harvestable in window) | ANY spring-growable can appear in Spring Crops, incl. year-2-ish and modded, at low weight |
| Quality crops | same, gold-star asks | not always vanilla's fixed four |
| Fish | `Data/Fish` + location/season/weather spawn data | condition windows respected (see season-gate rule) |
| Seasonal forage | `Data/Locations` forage tables **per season** | see the seasonal-forage ruling below |
| Monster drops | monster data (`Data/Monsters` drop tables) | Adventurer's/Slayer's: squid ink, crab, essences, slime, bat wings, bone fragments, desert-cave drops — anything a mob drops |
| Smeltables/bars | object categories | **iridium bar joins the bar-bundle pool** (rare roll) |
| Artisan goods | machine-output categories | wine one loop, aged roe the next |
| Minerals (geode) | geode-content data, gems excluded | for the Mineral Bundle |
| Artifacts | museum-donatable artifact set | for the Artifact Bundle |
| Books | bookseller-purchasable books, year-1 vetted | for the Book Bundle |

**Modded-content rules (user requirement):** unknown items get conservative default weights;
obtainability (season windows, spawn conditions) is read from the item's own data, never from
hardcoded vanilla lists; one **exclude-list** (config-extensible) blocks quest items and
unobtainables; `ItemRarityResolver` (price-threshold based) already handles modded items for JP.
**Verification pass with SVE enabled is a required test** (SVE sits installed-disabled on the PC
for exactly this).

### Seasonal foraging bundles — REVISED RULING 2026-07-14

The four seasonal foraging bundles KEEP their season identity (no any-season mixing — user
rejected that). Each season's pool = vanilla's forage for that season **plus curated harder
additions that fit the season**, e.g.:

- Spring += Common Mushroom, Red Mushroom (cave/Secret Woods).
- Fall/Winter += Purple Mushroom.
- Desert forage (Coconut, Cactus Fruit) as rare rolls where season-appropriate.
- **Large-quantity asks** as a roll type: e.g. Salmonberry x99, big berry/mushroom stacks —
  the "150 Salmonberries" idea at baseline scale.
- No duplicate items within a bundle.

### Stacks & qualities

One tunable config block; baseline ≈ vanilla feel (mostly stack 1-5, normal quality, occasional
silver/gold), plus the large-quantity forage rolls above. 0.13.0's tiers scale these knobs.

### Season-gate safety rule (generalizes the Four Seasons Sampler concern)

The user's catch: a bundle whose items can only be obtained late (e.g. winter forage) must not
be demanded by an earlier season's checkpoint. **Rule: the generator emits each bundle's
checkpoint quota ramp such that no season's cumulative quota exceeds the number of that
bundle's slots obtainable by the end of that season** (item season windows are known at
generation time). This is enforced for EVERY bundle, killing the "impossible gate" class the
same way the manifest kills the classification class.

## Vault — engine-owned

The user rejected leaving the Vault vanilla. At 0.12.0's single baseline difficulty the four
money bundles keep their structure with amounts **+25%** (2,500→3,125g, 5,000→6,250g,
10,000→12,500g, 25,000→31,250g). Hard's +50%-or-x2 lands with tiers in 0.13.0.
`VaultBundleMap` already derives indices from live `BundleData`, so it keeps working.

## Difficulty targets (user ruling 2026-07-14)

- **Easy** (0.13.0): achievable by a very skilled player in 1 loop.
- **Normal**: NOT achievable in 1 loop, even by a very skilled player.
- **Hard** (0.13.0): takes anyone many loops and upgrades to unlock/finish.

**0.12.0 ships a single difficulty tuned to the NORMAL bar.** Concretely: pool weights,
large-quantity rolls, and quality asks are tuned so that a khauser13-class player needs at
least one loop. Tuning happens in playtest against exactly that bar; the knobs all live in the
stacks/qualities/weights config block. (This supersedes the June "vanilla-equivalent baseline"
note — the driving feedback made clear vanilla-equivalent is too easy.)

## Testing

- **Unit (Core, pure):** generator determinism (same seed → identical set), pool derivation
  from synthetic data tables (incl. mod-shaped items), exclude-list, no-duplicates-per-bundle,
  season-gate quota rule (no season demands more than obtainable), vault +25% math, manifest ↔
  written BundleData consistency.
- **Integration (PC):** create run → verify generated set in the CC menu; `tly_failreset` →
  verify a DIFFERENT set next loop and the same set on replaying the reset; full donate/gate
  cycle on generated bundles; season-goals menu renders stacks/qualities.
- **SVE pass (required):** enable SVE, generate, confirm SVE items appear appropriately in
  pools, nothing unobtainable is demanded, no classification errors in the log.
- Suite baseline 563 green before/after; new tests added per component.
